# 핸드오프 아티팩트 양식 (Handoff Artifact)

> **왜 이게 골격인가?** 하네스가 "어떤 에이전트가 들어와도 동일하게" 굴러가는 비결은 **컨텍스트를
> 합치지 않고 끊되, 구조화된 아티팩트로 다음 역할이 정해진 상태에서 이어받는 것**입니다([team-roles
> §3](team-roles.md)). 이 파일이 그 아티팩트의 **단일 양식** — 한 작업당 이 한 장을 채워 넘깁니다.
> 매 핸드오프는 "이전 에이전트의 머릿속"이 아니라 이 문서를 읽고 시작합니다.

## 쓰는 법

1. 한 작업(기능/버그/슬라이스)마다 이 양식을 복사해 한 파일로 둔다 — 기본 위치는 프로젝트 루트
   `handoffs/handoff-<작업이름>.md`(폴더가 없으면 만든다). **위임할 때 이 파일 경로를 반드시 함께
   넘긴다** — 컨텍스트가 리셋된 다음 역할이 읽을 유일한 입력이므로.
2. 각 역할은 **자기 섹션만** 채운다(다른 섹션은 건드리지 않음 — 경계). 쓰기 도구가 없는
   역할(플래너·평가자)의 산출은 오케스트레이터가 받아 대신 기록한다.
3. 다음 역할은 이 문서 + 코드만 보고 시작한다. 막히면 `## 열린 질문`에 적고 멈춘다(추측 금지).
4. 상태는 `## 상태 로그`에 한 줄씩 누적(누가·언제·무엇). 되돌릴 수 있게.

---

## 작업 헤더

- **작업 이름:** <짧고 고유하게>
- **한 줄 목표:** <이 작업이 끝나면 사용자가 무엇을 할 수 있나>
- **현재 단계:** `PLAN | BUILD | EVAL | DONE | LEARN | BLOCKED`
- **담당(현재):** <역할/에이전트>

## 1. 계획 (플래너가 채움)

- **제품 맥락:** <왜 이게 필요한가, 누구를 위한 것인가>
- **상위 설계:** <접근 방식 — 세부 구현 아님>
- **기능 목록(빌드 단위):** 한 번에 하나씩 만들 수 있게 쪼갠 체크리스트
  - [ ] <기능 1>
  - [ ] <기능 2>
- **수용 기준(평가자가 채점할 기준):** <"무엇이 참이어야 통과인가" — 구체적·검증 가능>
- **범위 밖(하지 말 것):** <경계>
- **검증 방법:** <어떤 테스트/명령/관찰로 확인하나>

## 2. 빌드 (워커/제너레이터가 채움)

- **건드린 범위:** <파일/모듈 — 좁게>
- **한 일(기능별):** 계획의 각 기능에 대해 무엇을 했는지 한 줄
  - <기능 1> → <한 일>
- **결정/가정:** <비자명한 선택과 이유>
- **검증 실행 결과:** <돌린 명령 + 결과. 안 돌렸으면 "미실행"이라고 적기>
- **다음 역할에 전달:** <평가자가 특히 봐야 할 곳>

## 3. 평가 (평가자가 채움 — 만든 자와 반드시 다른 에이전트)

- **판정:** `PASS | FAIL`
- **수용 기준 대조:** 계획의 각 기준에 충족/미충족 + 근거
  - <기준 1> → 충족? <근거(실제로 돌려보거나 확인한 것)>
- **구체적 피드백(FAIL이면 제너레이터가 바로 고칠 수 있게):**
  - <무엇이·왜 잘못됐나 + 어디>
- **반복 횟수:** <이번이 몇 번째 BUILD↔EVAL 사이클인지>

## 4. 리뷰·병합 (선택 — 리뷰어/사람)

- **병합 전 점검:** <전체 맥락에서 본 리스크>
- **승인:** <누가·언제>

## 5. 학습 (큐레이터가 채움 — DONE 후, 비차단)

- **통한 전략(강화):** <다음에 더 자주 쓸 것>
- **교정·실패(약화/래칫):** <약화할 전략 / 규칙 한 줄로 박을 실패>
- **재사용 절차:** <이 작업에서 추출한 스킬/절차 — 없으면 "없음">
